# Health Analytics

Use this skill when the user uploads a new Apple Health or Weltory ZIP export, asks to update the health database, asks to sync workouts from Notion, or asks health questions that should be answered from the local database instead of memory.

## Fixed paths

Assume the OpenClaw workspace is the current working directory unless configured otherwise.

Use these paths inside the workspace:

- database: `data/health/health.duckdb`
- imports: `data/health/imports`
- extracted: `data/health/extracted`
- logs: `data/health/logs`
- cache: `data/health/cache`

## Rules

- Never claim data was imported unless the import script completed successfully.
- Never rebuild the database if it already exists unless the user explicitly asks.
- Always ingest incrementally.
- Deduplicate by native source record id when present.
- If a native id is missing, deduplicate by `timestamp + metric + source`.
- Always append an audit record to the import log.
- Prefer DuckDB for storage and SQL.
- Prefer Python for parsing.
- When answering health questions, query the database first.
- Use exact dates, values, and source names in answers.
- State uncertainty when data is incomplete.
- Do not give medical advice as if it were diagnosis; frame improvement suggestions as observations and hypotheses grounded in the user's data.

## Supported sources

- Apple Health ZIP export

- Notion workout database sync

## Ingestion workflow

When the user uploads a ZIP and asks to import or update:

1. Ensure the database exists by running `python scripts/init_db.py` from the skill directory or an equivalent absolute path.
2. Copy the ZIP into `data/health/imports/`.
3. Run `python scripts/ingest.py --zip <path-to-zip>`.
4. Read the script output.
5. Summarize:
   - source detected
   - files found
   - rows inserted
   - rows skipped
   - date range covered
6. Tell the user what questions can now be answered.

## Update behavior

Each new ZIP can include old data plus new data. The skill must:

- preserve the existing database
- import only new records
- keep a complete audit log
- avoid duplicate rows across repeated imports

## Notion sync workflow

When the user asks to sync workout series from Notion:

1. Check whether `NOTION_API_KEY` and `NOTION_DATABASE_ID` are present.
2. Run `python scripts/sync_notion_workouts.py`.
3. Upsert results into a `notion_workouts` table using the Notion page id as the primary key.
4. Summarize records synced and date range.
5. If credentials are missing, say so clearly and do not pretend the sync ran.

## Query workflow

When the user asks a health question:

1. Run `python scripts/query.py --question "<user-question>"`.
2. Base the answer on returned SQL results, not on memory.
3. Include concrete values, dates, and trends.
4. If useful, include a short interpretation.
5. Separate observed correlations from hypotheses.

## High-priority questions

The skill should handle these well:

- resting heart rate trend
- sleep duration and sleep consistency
- relationship between sleep and energy
- relationship between sleep and next-day heart rate or intervals
- medication adherence and downstream effects
- exercise volume and intensity over time
- steps and calories trends
- anomaly detection
- what can I do to improve sleep and energy

## Improvement-question handling

For questions like "what can I do to improve sleep and energy":

1. Query recent sleep, exercise, heart rate, intervals, medication adherence, and workout load.
2. Identify patterns tied to worse and better days.
3. Separate observed correlations from hypotheses.
4. Provide practical suggestions grounded in the user's data.
5. Avoid overstating causality.

## Suggested core tables

The scripts should maintain tables like these:

- `import_log`
- `sleep_daily`
- `heart_rate_samples`
- `heart_rate_intervals`
- `steps_daily`
- `calories_daily`
- `exercise_sessions`
- `medications`
- `notion_workouts`

## Dedupe policy

- Primary dedupe: native record id + source
- Fallback dedupe: timestamp + metric + source
- Keep audit rows even when all data rows were duplicates

## Failure handling

- If the ZIP format is unknown, stop and explain why.
- If a parser fails, log the failure and report the affected source/file.
- If a query cannot be answered from current data, say what is missing.

## Security and caution

- Do not expose tokens or secrets in chat output.
- Do not run destructive shell commands unless the user explicitly asks.
- Do not delete prior imports automatically.
